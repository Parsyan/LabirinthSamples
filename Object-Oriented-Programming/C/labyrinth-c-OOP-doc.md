# labyrinthByOOP.c — բացատրություն

Այս փաստաթուղթը նկարագրում է, թե ինչպես է կառուցված `labyrinthByOOP.c`-ն, ինչու հենց այդպես, և ինչպես են աշխատում ալգորիթմները։
Ենթադրվում է, որ դու արդեն կարդացել ես պրոցեդուրային տարբերակը (`labyrinthByPP.c`), ուստի խաղի **տրամաբանությունն** արդեն ծանոթ է։ Այստեղ մենք կենտրոնանում ենք միայն այն հարցի վրա, թե **ինչպես է OOP-ը հայտնվում C-ում**, որտեղ classes-ներ ընդհանրապես չկան։

---

## 1. Ընդհանուր գաղափարը

Խաղի կանոնները չեն փոխվել՝
- 10×20 տախտակ,
- `@` խաղացողը պետք է հասնի `&` դրոշակին,
- `#` ժայռերը խանգարում են,
- կառավարումը՝ `w/a/s/d`։

Փոխվել է միայն **կոդի կազմակերպումը**։ PP-տարբերակում մենք ունեինք.

- մի շարք ֆունկցիաներ,
- `char board[H][L]` երկչափ զանգված,
- մի քանի առանձին փոփոխական՝ խաղացողի և դրոշակի դիրքի, ինչպես նաև հաղթանակի դրոշի համար։

OOP-տարբերակում նույն խաղն ապրում է մի քանի **«դաս»**ի մեջ (C-ում՝ struct-ով նմանակված), և տախտակը ոչ թե նիշերի, այլ **օբյեկտների** ցանց է.

```
Cell (աբստրակտ բազային)
 ├── EmptyCell   (' ')
 ├── Rock        ('#')
 ├── Flag        ('&')
 └── Player      ('@')

Board   — պահում և կառավարում է Cell** ցանցը
Game    — վարում է խաղի loop-ը, ձեռնարկը, մուտքը
main    — մուտքի կետ, ստեղծում է Game ու կանչում
```

Ընդհանուր հոսքը նույնն է, ինչ PP-ում.

```
ձեռնարկ ցույց տալ  →  Board կառուցել  →  տպել  →
   ↓
   while չհաղթել:
       ուղղություն խնդրել  →  շարժվել  →  տպել  →  ստուգել՝ հաղթե՞լ ենք
```

Բայց յուրաքանչյուր քայլ արդեն ոչ թե «ֆունկցիա է, որ ստանում է զանգված ու state», այլ «մեթոդ» է որևէ struct-ի վրա, որը գիտի իր սեփական տվյալների մասին։

## 2. OOP C-ում՝ առանց class-ի, «ձեռքով»

Java-ում կամ Python-ում OOP-ը լեզվի մաս է. գրում ես `class`, `extends`, `@Override`, և կոմպիլյատորը հոգ է տանում մնացածի մասին։ C-ում **այս բառերը չկան**։ Բայց OOP-ի գաղափարը կարող է գործնականում նույնքան լավ աշխատել, պարզապես պետք է հենց մեզ վրա վերցնենք այն, ինչ Java-ն ինքն է անում։ Ահա երեք գործիք, որոնք C-ում օգտագործում ենք.

### 2.1 «class»-ը՝ որպես `struct`

C-ի `struct`-ը մի տուփ է, որի մեջ տվյալներ կան։ Մենք պարզապես ավելացնում ենք **առանձին ֆունկցիաներ**, որոնք առաջին պարամետրով ստանում են այդ struct-ի ցուցիչը.

```c
typedef struct {
    Cell ***grid;
    short height, length;
    Cell *player, *flag;
    int won;
} Board;

static void board_print_world(const Board *self);
static void board_move_up(Board *self);
```

`self` պարամետրն այստեղ նույնն է, ինչ Java-ի `this`-ը։ `board_move_up(b)` կարդա որպես `b.moveUp()`։ Ֆունկցիաների անունները սկսվում են `board_`-ով, որպեսզի ակնհայտ լինի, թե որ «դասի» մեթոդն է։

### 2.2 «inheritance»-ը՝ struct-ի **առաջին դաշտով**

C-ում միակ ճանապարհը, որ մի struct «ժառանգի» մյուսի դաշտերը, սա է. **առաջին դաշտով** ներդնել ծնող struct-ը.

```c
typedef struct {
    Cell base;  /* ⭐ ՊԱՐՏԱԴԻՐ ԱՌԱՋԻՆ դաշտ */
} Rock;
```

Ինչու հենց առաջինը։ Որովհետև C-ի ստանդարտը երաշխավորում է. struct-ի առաջին դաշտի հասցեն և struct-ի սկզբի հասցեն **նույնն են**։ Այսինքն՝ եթե ունենք `Rock *r`, ապա `(Cell *) r`-ը վավեր ցուցիչ է դեպի `r->base`։ Board-ը կարող է իր `grid[i][j]`-ում պահել «Cell *» և չիմանալ, թե իրականում ներքևում Rock է, Flag, թե Player. ինչպես C-ի կաստինգը ճիշտ կաշխատի։

Սա մեր C-ով գրած տարբերակն է հենց այն մտքի, որ «Rock-ը *is-a* Cell»։

### 2.3 «virtual method»-ը՝ vtable-ով (function pointer-ների աղյուսակ)

Java-ում երբ կանչում ես `cell.getSymbol()`, JVM-ն ինքն է որոշում, թե որ ենթադասի մեթոդը կանչել։ C-ում մենք պետք է այս «աղյուսակը» **ձեռքով** կառուցենք։ Կոչվում է **vtable** (virtual method table)։

```c
typedef struct CellVTable {
    char (*get_symbol)(const struct Cell *self);
    int  (*is_blocking)(const struct Cell *self);
    void (*destroy)(struct Cell *self);
} CellVTable;

typedef struct Cell {
    const CellVTable *vt;   /* որ ենթատեսակի աղյուսակն է */
    int row;
    int col;
} Cell;
```

Ամեն Cell-ի մեջ պահում ենք **մի ցուցիչ** դեպի այդ տեսակին համապատասխան vtable։ Աղյուսակում գրված են ֆունկցիաները, որոնք պետք է կանչվեն այդ տեսակի համար։ Կանչը կատարվում է այսպես.

```c
static char cell_get_symbol(const Cell *self) {
    return self->vt->get_symbol(self);
}
```

Կարդա այսպես. «Հարցրու cell-ին, թե իր սեփական աղյուսակում որն է `get_symbol`-ի ֆունկցիան, ու կանչիր այն»։ Հենց սա է **polymorphism**-ը C-ում. նույն կանչը, տարբեր արդյունք՝ կախված նրանից, թե vt-ն որ աղյուսակին է ցույց տալիս։

### Ինչու հենց այսպես

Կարող էինք գրել՝

```c
if (cell->type == ROCK) ...
else if (cell->type == FLAG) ...
```

Բայց այդ դեպքում **ամեն տեղ**, որտեղ որևէ տարբերություն կա ենթատեսակների միջև, պետք է կրկնեինք այդ `if/else`-երը։ Նոր տեսակ ավելացնելու համար պետք կլիներ գտնել բոլոր այդ տեղերը։ vtable-ով՝ նոր տեսակը պարզապես ավելացնում ենք իր ֆայլի մեջ, մի նոր vtable կառուցելով, և մնացած կոդը չենք էլ դիպչում։

## 3. OOP-ի չորս սկզբունքները՝ այս կոդում

Երբ խոսում ենք OOP-ի մասին, սովորաբար նշում են չորս հիմնական գաղափար. **Abstraction**, **Encapsulation**, **Inheritance**, **Polymorphism**։ Տեսնենք յուրաքանչյուրը այս C կոդում որտեղ է հայտնվում։

### 3.1 Abstraction (վերացարկում)

**Գաղափարը.** թաքցնել ավելորդ մանրուքները և ցույց տալ միայն այն, ինչ կարևոր է։

`Cell`-ի «ինտերֆեյսը» (այն, ինչ նա խոստանում է) ընդամենը երեք գործողություն է.

```c
char cell_get_symbol(const Cell *self);
int  cell_is_blocking(const Cell *self);
void cell_destroy(Cell *self);
```

Մենք ասում ենք. «Տախտակի վրա կա ինչ-որ բան, որը ունի որևէ խորհրդանիշ, կարող է փակել կամ չփակել ճանապարհը, և կարող է մաքրել իր հիշողությունը»։ Մենք **չենք բացում**, թե ինչ է ներքևում՝ ժայռ, դրոշակ, թե խաղացող։ Board-ի համար բավական է, որ կարող է կանչել այս երեքը։

### 3.2 Encapsulation (պատամփակում)

**Գաղափարը.** օբյեկտի ներքին տվյալները թաքցնել, իսկ դրսից թույլատրել միայն ստուգված գործողություններ։

C-ում `private` բառ չկա, բայց ունենք երկու գործիք.

- **`static` ֆունկցիաներ** — ֆայլից դուրս չեն երևում։ Մեր կոդում **գրեթե բոլոր** ֆունկցիաները `static` են (`board_place_rocks`, `board_path_exists`, `board_try_move`, և այլն)։ Արտաքին աշխարհը (եթե այս ֆայլը մի օր լիներ մեծ նախագծի մաս) տեսներ միայն `main`-ը։
- **Անվանման կարգապահություն** — եթե գրում ես `b->grid[5][3] = ...` ինչ-որ ուրիշ տեղ, դա «սխալ» է. ուղղում ենք միայն `board_*` ֆունկցիաներով։ C-ն չի ստիպելու, բայց պայմանավորվածությունը պարզ է։

Համեմատիր PP-տարբերակի հետ. այնտեղ `board[][]` զանգվածն ու state-ը շրջում էին որպես պարամետրեր, և ցանկացած ֆունկցիա կարող էր դրանք ինչ ուզում, փոխեր։ Հիմա, որպեսզի որևէ բան փոխվի Board-ում, պետք է անցնել Board-ի մեթոդով։

### 3.3 Inheritance (ժառանգում)

**Գաղափարը.** մի «դասը» կարող է վերցնել մյուսի ամբողջ դաշտերը և ավելացնել կամ վերագրել մի մաս։

C-ում սա արվում է struct-ի առաջին դաշտով.

```c
typedef struct { Cell base; } EmptyCell;
typedef struct { Cell base; } Rock;
typedef struct { Cell base; } Flag;
typedef struct { Cell base; } Player;
```

Չորս ենթատեսակները բոլորն ունեն Cell-ի `vt`, `row`, `col` դաշտերը՝ առանց դրանք կրկնելու։ «Ժառանգում են»՝ ոչ թե բառերով, այլ հիշողության դասավորությամբ։

### 3.4 Polymorphism (բազմաձևություն)

**Գաղափարը.** նույն մեթոդի կանչը կարող է վարվել տարբեր ձևերով՝ կախված օբյեկտի իրական տեսակից։

Տես `board_print_world`-ը.

```c
printf("%c", cell_get_symbol(self->grid[i][j]));
```

Կոդը **չի հարցնում** «ի՞նչ ես՝ ժայռ, դրո՞շ, թե խաղացող»։ Նա պարզապես ասում է` «արտածիր քո խորհրդանիշը»։ Թե որ ֆունկցիան իրականում կկանչվի, որոշում է `self->vt`-ն այդ պահի դրությամբ։

Նույն կերպ `board_path_exists`-ի մեջ.

```c
if (cell_is_blocking(self->grid[nr][nc])) continue;
```

Չենք գրում «եթե Rock է, շրջանցիր»։ Հարցնում ենք բուն օբյեկտին։ Եթե վաղը ավելացնենք նոր տեսակի խոչընդոտ (օրինակ՝ «ջուր»), որի vtable-ում `is_blocking`-ը նույնպես վերադարձնում է 1, **այս ոչ մի տող փոխել պետք չի լինի**։

## 4. `Cell`-ի հիերարխիան մանրամասն

### 4.1 `CellVTable` և `Cell` բազային struct-ները

```c
typedef struct CellVTable {
    char (*get_symbol)(const struct Cell *self);
    int  (*is_blocking)(const struct Cell *self);
    void (*destroy)(struct Cell *self);
} CellVTable;

typedef struct Cell {
    const CellVTable *vt;
    int row;
    int col;
} Cell;
```

- **`vt`** — ցույց է տալիս այս ենթատեսակին պատկանող vtable-ին։ `const`-ն այստեղ նշանակում է, որ vtable-ի բովանդակությունը չենք փոխի (և չպետք է փոխենք. աղյուսակը նույնն է բոլոր ժայռերի համար)։
- **`row, col`** — ընդհանուր դաշտեր բոլոր cell-ների համար։ Ոչ մեկը չպետք է կրկնի դրանք իր ենթատեսակում։

### 4.2 Ընդհանուր (default) մեթոդները

Որոշ բաներ նույնն են բոլոր ենթատեսակների համար, և իմաստ չունի կրկնել.

```c
static int default_not_blocking(const struct Cell *self) {
    (void) self;
    return 0;
}

static void default_destroy(struct Cell *self) {
    free(self);
}
```

- **`default_not_blocking`** — մեծ մասը cell-ները չեն փակում ճանապարհը։ Միայն Rock-ն ունի իր սեփական `is_blocking`։
- **`default_destroy`** — մեր բոլոր ենթատեսակները մեկ struct են՝ առանց ներքին ցուցիչների, ուստի `free`-ն բավական է։ Եթե հետագայում Player-ի մեջ ավելացնենք, օրինակ, ինվենտար, որը նույնպես պետք է մաքրվի, կարող ենք Player-ին տալ սեփական `destroy`։

`(void) self;` տողը պարզապես կոմպիլյատորին ասում է` «գիտեմ, որ `self`-ը չեմ օգտագործում, թողի, որ զգուշացում չտա»։

### 4.3 EmptyCell — դատարկ վանդակ

```c
static char empty_get_symbol(const struct Cell *self) {
    (void) self;
    return ' ';
}

static const CellVTable EMPTY_VTABLE = {
    empty_get_symbol,
    default_not_blocking,
    default_destroy
};

static Cell *empty_new(int row, int col) {
    EmptyCell *e = (EmptyCell *) malloc(sizeof(EmptyCell));
    e->base.vt  = &EMPTY_VTABLE;
    e->base.row = row;
    e->base.col = col;
    return (Cell *) e;   /* անվտանգ. base-ը առաջին դաշտն է */
}
```

Երեք բան է կատարվում.

1. **vtable** — մեկ ընդհանուր կառուցվածք բոլոր EmptyCell-երի համար. ուստի `static const`։ Ամեն դատարկ վանդակ ընդամենը ցույց է տալիս սրան։
2. **«կոնստրուկտոր» `empty_new`** — հատկացնում է հիշողություն, լրացնում է vt-ն և սկզբնական դիրքը, վերադարձնում է «Cell *»` որպեսզի դրսից օգտագործողը չիմանա, թե իրականում ներքևում EmptyCell է։
3. **(Cell *) cast-ի անվտանգությունը** — հենց որ `base`-ը առաջին դաշտն է, այս cast-ը երաշխավորված է, որ ճիշտ կաշխատի։

### 4.4 Rock — փակում է ճանապարհը

```c
static char rock_get_symbol(const struct Cell *self) { (void) self; return '#'; }
static int  rock_is_blocking(const struct Cell *self) { (void) self; return 1; }

static const CellVTable ROCK_VTABLE = {
    rock_get_symbol,
    rock_is_blocking,    /* սա է տարբերությունը */
    default_destroy
};
```

Rock-ը միակն է, որ վերագրում է `is_blocking`-ը։ Մնացած ամեն ինչ նույնն է, ինչ EmptyCell-ում. տարբեր vtable, նույն մոտեցում։

### 4.5 Flag, Player — միայն խորհրդանիշով են տարբեր

```c
static char flag_get_symbol(const struct Cell *self)   { return '&'; }
static char player_get_symbol(const struct Cell *self) { return '@'; }
```

Player-ի համար հատկապես ոչինչ առանձնահատուկ չկա այս փուլում. պարզապես ուրիշ խորհրդանիշ։ Բայց **հենց իմաստն** այն է, որ եթե հետագայում ուզենաս Player-ին տալ «քայլերի հաշվիչ» կամ «կյանք»-ի կետեր, ուղղակի լրացնում ես Player struct-ը, և մյուս դասերը չեն փոխվում։

### 4.6 «instanceof»-ը C-ում՝ vtable-ի համեմատությամբ

Java-ի `instanceof`-ի փոխարեն գրում ենք.

```c
static int is_flag(const Cell *c)  { return c->vt == &FLAG_VTABLE; }
static int is_empty(const Cell *c) { return c->vt == &EMPTY_VTABLE; }
```

Ինչու։ Քանի որ ամեն ենթատեսակ ունի **մի սեփական vtable** (և բոլոր նրա օրինակները ցույց են տալիս նույն vtable-ին), vtable-ի հասցեն իրականում խաղում է type id-ի դեր։

Կարևոր է հասկանալ՝ սա մեզ պետք է **ուղղակի այն տեղերում**, որտեղ տիպային ստուգումը խաղի կանոնի մասն է (օրինակ՝ «արդյոք այստեղ դրոշակն է»)։ Մնացած դեպքերում, որտեղ բավական է «արդյոք փակվա՞ծ է» հարցը, vtable-ով կանչն ավելի մաքուր է` `cell_is_blocking(...)`։
