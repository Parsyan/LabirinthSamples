# Labyrinth.java — բացատրություն

Այս փաստաթուղթը նկարագրում է, թե ինչպես է կառուցված `Labyrinth.java`-ն, ինչու է հենց այդպես կառուցված, և ինչպես են աշխատում ալգորիթմները։ Ենթադրվում է, որ դու արդեն գրել ես Tic Tac Toe, ուրեմն ծանոթ ես երկչափ զանգվածներին, `Scanner`-ին, `while`/`if`-երին և մեթոդներին։

---

## 1. Ընդհանուր գաղափարը

Labyrinth-ը (լաբիրինթոս) Tic Tac Toe-ից մի փոքր ավելի բարդ խաղ է հետևյալ պատճառներով.

1. **Տախտակն ավելի մեծ է** — ոչ թե 3×3, այլ 10×20։
2. **Խաղացողը շարժվում է** — Tic Tac Toe-ում դու դնում ես խորհրդանիշ ու վերջ, իսկ այստեղ `@`-ն ամեն քայլից հետո փոխում է դիրքը։
3. **Տախտակը պատահական է գեներացվում** — ամեն խաղ տարբեր է լինում։
4. **Կա կանոն, որ պետք է երաշխավորել** — խաղացողը պետք է կարողանա հասնել դրոշակին, այսինքն ժայռերը չպետք է փակեն ճանապարհը։

Խաղի վերջնական հոսքը՝

```
ձեռնարկ ցույց տալ  →  տախտակ գեներացնել  →  տպել  →
   ↓
   while չհաղթել:
       ուղղություն խնդրել  →  շարժվել  →  տպել  →  ստուգել՝ հաղթե՞լ ենք
```

## 2. Ինչու ենք state-ը պահում `int[]`-ի մեջ

Բայց Java-ում կա մեկ կարևոր կանոն. եթե մեթոդին `int` ես փոխանցում, դա **պատճենով** է փոխանցվում (pass-by-value)։ Այսինքն եթե `moveUp()`-ի ներսում փորձես փոխել `playerRow`-ը, արտաքին փոփոխականը չի փոխվի։

Լուծումը՝ բոլոր փոփոխվող արժեքները փաթաթել մեկ զանգվածի մեջ։ Զանգվածը **հղումով** է փոխանցվում (reference), ուրեմն ցանկացած մեթոդ կարող է փոխել նրա պարունակությունը, ու փոփոխությունը երևում է դրսից.

```java
int[] state = new int[5];
// state[0] = playerRow
// state[1] = playerCol
// state[2] = flagRow
// state[3] = flagCol
// state[4] = won (0 = ոչ, 1 = այո)
```

Սա ամենագեղեցիկ լուծումը չէ (ավելի կոկիկ կլիներ առանձին `class`-ով, բայց դա արդեն OOP-ի թեմա է), բայց աշխատում է ու հասկանալի է։

`Scanner`-ն ու `Random`-ը նույնպես օբյեկտներ են (հղումով), ուստի դրանք պարզ լոկալ փոփոխականներով ենք փոխանցում։

## 3. `createWorldBoard` — տախտակի կառուցումը

```java
public static char[][] createWorldBoard(short height, short length, int[] state, Random random)
```

Քայլերը՝

1. **Ստեղծել դատարկ տախտակ** — բոլոր վանդակները լցնել `' '` (բացատ) նիշով։ Սա մեր «անցանելի տարածությունն» է։
2. **Դնել խաղացողին `@`** — պատահական տողի և սյան վրա։
3. **Դնել դրոշակը `&`** — պատահական դատարկ վանդակում (loop-ով, որ չհամընկնի խաղացողի հետ)։
4. **Կանչել `placeRocks`-ը** — ժայռերը դնելու համար։

Ուշադրություն դարձրու, որ խաղացողն ու դրոշակը դրվում են **առաջ**, ժայռերը՝ **հետո**։ Դա կարևոր է, որովհետև ժայռեր տեղադրելիս մենք ստուգում ենք՝ խաղացողից մինչև դրոշակ ճանապարհ դեռ կա՞։ Եթե դրանք դեռ չկան, ստուգելու բան էլ չկա։

## 4. `placeRocks` — ինչու ոչ պարզ պատահական տեղադրում

Միամիտ տարբերակը կլիներ՝

> Պատահական ընտրիր 10 դատարկ վանդակ, դիր `#`։

Բայց դա կարող է բերել ուր որ խաղացողը շրջապատված է ժայռերով ու չի կարող շարժվել, կամ դրոշակը մեկուսացված է։ Խաղն այդ դեպքում անհնար է լինում։

Ուստի մենք օգտագործում ենք **«փորձիր ու հետ կանչիր»** ալգորիթմը (trial and rollback)՝

```
placed = 0
while placed < 10:
    ընտրիր պատահական դատարկ վանդակ (r, c)
    դիր այնտեղ '#'
    եթե դեռ կա ճանապարհ @ -> &:
        լավ է, placed++
    հակառակ դեպքում:
        հետ վերցրու '#'-ը, դարձրու բացատ
```

Ինչու է սա աշխատում.

- Ամեն ժայռ դնելուց հետո մենք **երաշխավորում ենք**, որ ճանապարհը դեռ կա։
- Եթե ինչ-որ ժայռ կկտրի ճանապարհը, պարզապես չենք դնի այնտեղ ու կփորձենք այլ վայր։
- Քանի որ սկզբում դաշտը գրեթե դատարկ էր, ինվարիանտը (ճանապարհի առկայությունը) պահվում է ողջ ընթացքում։

Ավելացրել ենք նաև `attempts < 1000` պաշտպանություն, որ եթե ինչ-ինչ պատճառով 10 ժայռ դնելու տեղ չլինի (շատ փոքր տախտակ), `while`-ը անվերջ չքաշվի։

## 5. `pathExists` — ինչպես ենք ստուգում ճանապարհը (DFS)

Սա հավանաբար ամենաանծանոթ կտորն է՝ համեմատած Tic Tac Toe-ի հետ։ Մենք պետք է պատասխանենք հարցին.

> Տվյալ տախտակում խաղացողի դիրքից մինչև դրոշակը կա՞ ճանապարհ՝ չանցնելով ժայռերի վրայով։

Սա կոչվում է **գրաֆի որոնում**։ Տախտակը կարող ենք ընդունել որպես գրաֆ՝ ամեն վանդակ՝ հանգույց, հարևան վանդակները (վերև/ներքև/ձախ/աջ)՝ կողեր։ Ժայռերը՝ արգելված հանգույցներ։

Մենք օգտագործում ենք **DFS** (Depth-First Search՝ խորությամբ որոնում)՝ stack-ով.

```
stack-ում դիր սկզբնական դիրքը
visited-ում նշիր այն
while stack-ը դատարկ չէ:
    վերցրու վերևի վանդակը
    եթե դա դրոշակն է → true
    իր 4 հարևաններից յուրաքանչյուրի համար:
        եթե բացատ է կամ '&' է, ու դեռ չենք այցելել → ավելացրու stack-ին
return false
```

Ինչու DFS, ոչ թե BFS. երկուսն էլ կաշխատեն հենց այս խնդրի համար (մեզ կարճ ճանապարհ պետք չէ, միայն գոյությունն ենք ստուգում)։ DFS-ը մի փոքր ավելի պարզ է stack-ով, ուստի ընտրել ենք դա։

Ինչու չենք օգտագործել ռեկուրսիա. ռեկուրսիան ավելի կարճ կլիներ, բայց երբ դեռ սովորում ես, ձեռքով stack-ով տարբերակն ավելի թափանցիկ է՝ տեսնում ես, թե ինչ է կատարվում քայլ առ քայլ։

**Կարևոր մանրուք** — `visited` զանգվածը. առանց դրա DFS-ը անվերջ պտտվում է երկու հարևան վանդակների միջև։

## 6. `moveUp/Down/Left/Right` + `tryMove`

Չորս ուղղության մեթոդները պարզապես կանչում են `tryMove`-ը՝ տալով նոր կոորդինատները։ Կոպիա-կպցնելու փոխարեն ընդհանուր տրամաբանությունն ապրում է `tryMove`-ի մեջ.

```java
public static void tryMove(char[][] board, int newRow, int newCol, int[] state)
```

Ստուգումները հերթականությամբ.

1. **Արդյոք դուրս է տախտակից** → մնում ենք տեղում, տպում ենք հաղորդագրություն։
2. **Արդյոք `#` ժայռ է** → մնում ենք տեղում, տպում ենք «A rock blocks your way»։
3. **Արդյոք `&` դրոշակ է** → տեղափոխվում ենք այնտեղ, դնում ենք `state[4] = 1` (հաղթեցինք)։
4. **Հակառակ դեպքում** (դատարկ վանդակ) → պարզ տեղափոխություն։

Տեղափոխությունը կատարվում է երեք քայլով՝
- հին վանդակը դարձնել `' '`,
- նոր վանդակը դարձնել `'@'`,
- `state[0]`-ն ու `state[1]`-ը թարմացնել։

## 7. `printWorld` — տպելու մասը

Tic Tac Toe-ից ծանոթ ձև է, պարզապես ավելի մեծ տախտակ։ Շրջանակ ենք գծել `+---+` և `|` նիշերով, որ տախտակի սահմանները տեսանելի լինեն. դա օգնում է խաղացողին հասկանալ, որ բացատները տախտակի մաս են, ոչ թե էկրանի դատարկ տարածք։

## 8. `inputDirection` — մուտքի վավերականացում

Tic Tac Toe-ում, հավանաբար, դու սովորել ես, որ օգտատերը կարող է անհավասարակշիռ բան մուտքագրել (օրինակ՝ `Enter` սեղմել առանց ոչինչ գրելու)։ Այստեղ նույն գաղափարն է՝ `while(true)` ցիկլում մուտք ենք խնդրում մինչև որ ստանանք `w`, `a`, `s`, կամ `d`-ից մեկը.

- Դատարկ տող → մերժում ենք։
- Չվավեր նիշ → մերժում ենք։
- `toLowerCase` → որ `W` գրելն էլ ընդունվի։
- `charAt(0)` → վերցնում ենք միայն առաջին նիշը, որ «www» գրողն էլ առաջ չհոգնեցնի։

Մեթոդը վերադարձնում է վավեր նիշը, ուստի ցանկացած կոդ, որ կանչում է սա, կարող է վստահ լինել, որ ստացած նիշը մեկն է չորսից։

## 9. `labyrinth` — հոսքի ուղեղը

Սա խաղի «main loop»-ն է, որը կապում է ամեն ինչը.

```java
int[] state = new int[5];
Scanner scanner = new Scanner(System.in);
Random random = new Random();

printUserManual();
char[][] board = createWorldBoard((short) 10, (short) 20, state, random);
printWorld(board);

while (state[4] == 0) {        // քանի դեռ չենք հաղթել
    char dir = inputDirection(scanner);
    if (dir == 'w') moveUp(board, state);
    else if (dir == 's') moveDown(board, state);
    else if (dir == 'a') moveLeft(board, state);
    else if (dir == 'd') moveRight(board, state);

    printWorld(board);

    if (state[4] == 1) {
        System.out.println("You reached the flag. You win!");
    }
}
```

Ուշադրություն դարձրու, որ `state`-ը, `scanner`-ը և `random`-ը **այստեղ են ստեղծվում**, ոչ թե դասի վերևում։ Դա էր սկզբնական պահանջը՝ դաշտերը տեղափոխել `labyrinth()`-ի մեջ։ `main()`-ը պարզապես կանչում է `labyrinth()`-ը՝

```java
public static void main(String[] args) {
    labyrinth();
}
```

## 10. Ինչու հենց այս կառուցվածքը

Մի քանի սկզբունք, որ արժի նկատել.

- **Մեկ մեթոդ՝ մեկ պատասխանատվություն**։ `createWorldBoard`-ը կառուցում է, `printWorld`-ը տպում է, `tryMove`-ը շարժում է։ Եթե ինչ-որ բան ինչ-որ տեղ սխալ է, գիտես՝ որտեղ նայել։
- **Ճանապարհի երաշխիք**։ Մենք չենք ապավինում «հուսանք, որ պատահականը լավը կլինի» սկզբունքին, այլ ամեն ժայռի դնելուց հետո ստուգում ենք։ Սա կոչվում է **invariant maintenance**։
- **State-ը մեկ տեղում**։ Քանի դեռ չենք սովորել OOP, `int[] state`-ը պարզ ձև է մի քանի կապված արժեքներ իրար հետ պահելու։ Հետո, երբ սովորենք `class`-եր, սա դառնում է `Player` ու `Game` օբյեկտներ։
- **Մուտքը պաշտպանված է**։ Տրամաբանությունը (շարժումը) երբեք չի տեսնում սխալ մուտք, որովհետև `inputDirection`-ը զտում է ամեն ինչ։

## 11. Հնարավոր հաջորդ քայլեր (եթե ուզում ես խաղալ)

- Ավելացնել քայլերի հաշվիչ — հաշվել, թե քանի քայլում հասար։
- Ավելացնել «quit» հրաման (օրինակ՝ `q` նիշը)։
- Տախտակի չափը խնդրել օգտատիրոջից։
- Ավելի շատ ժայռեր կամ մի քանի դրոշակ։
- Փոխարինել `int[] state`-ը իրական `class`-ով — սա կլինի քո առաջին հանդիպումը OOP-ի հետ։
