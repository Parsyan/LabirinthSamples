# labyrinth.py — բացատրություն

Այս փաստաթուղթը նկարագրում է, թե ինչպես է կառուցված `labyrinth.py`-ն, ինչու հենց այդպես, և ինչպես են աշխատում ալգորիթմները։ Java-ի տարբերակն արդեն ունենք (`Labyrinth.java` + `Labyrinth-doc.md`), ուստի այստեղ շեշտը կդնենք **Python-ին հատուկ բաների** վրա՝ ինչը հեշտացավ, ինչը մի փոքր այլ կերպ պետք էր մտածել, և ինչու։

---

## 1. Ընդհանուր գաղափարը

Խաղի կանոնները նույնն են, ինչ Java-ի տարբերակում.

1. 10×20 տախտակ, պատահական գեներացված։
2. Մեկ խաղացող `@`, մեկ դրոշակ `&`, 10 ժայռ `#`։
3. Շարժում՝ `w/a/s/d`։
4. Ժայռերը երբեք չեն կտրում խաղացողից մինչև դրոշակը հասնելու ճանապարհը։

Հոսքը.

```
ձեռնարկ ցույց տալ  →  տախտակ գեներացնել  →  տպել  →
   ↓
   while չհաղթել:
       ուղղություն խնդրել  →  շարժվել  →  տպել  →  ստուգել՝ հաղթե՞լ ենք
```

Python-ի և Java-ի միջև տարբերությունները հիմնականում **լեզվի առանձնահատկություններից** են գալիս, ոչ թե ալգորիթմից։ Ալգորիթմը նույնն է։

---

## 2. Ինչու ենք state-ը պահում `dict`-ի մեջ

Java-ի փաստաթղթում նկարագրեցինք, որ `int`-երը պատճենով են փոխանցվում, ուստի state-ը դրեցինք `int[]`-ի մեջ՝ որ հղումով գնա։

Python-ում նմանատիպ հարց կա, բայց մի փոքր տարբեր ձևակերպումով. **Python-ում ամեն ինչ object է, և անունները (variables) կապվում են object-ներին։** Անփոփոխ տիպերը (`int`, `str`, `tuple`) երբ վերագրում ես ֆունկցիայի ներսում, պարզապես լոկալ անունն է փոխվում, արտաքին փոփոխականը չի փոխվում.

```python
def f(x):
    x = 10        # միայն լոկալ `x`-ն է փոխվում
```

Բայց **փոփոխելի (mutable) օբյեկտները**՝ `list`, `dict`, `set`, ինչպես նաև `class`-ի instance-ները, կարող են ներսից փոփոխվել, և փոփոխությունը երևում է դրսից.

```python
def f(d):
    d["key"] = 10   # արտաքին dict-ն էլ է տեսնում փոփոխությունը
```

Ուստի state-ը պահում ենք `dict`-ի մեջ.

```python
state = {
    "player_row": 0,
    "player_col": 0,
    "flag_row":   0,
    "flag_col":   0,
    "won":        False,
}
```

Ինչու `dict`, ոչ թե `list` (ինչպես Java-ում `int[]`).

- **Կարդալիությունն ավելի լավն է**։ `state["player_row"]` շատ ավելի թափանցիկ է, քան `state[0]`։ Java-ում ստիպված էինք գրել մեկնաբանություններ, թե ինչ է նշանակում `state[0]`, `state[1]` և այլն։ Python-ում անունները բուն կոդի մեջ են։
- **Պակաս սխալներ**։ Չես կարող դիպչիր սխալ ինդեքսին (օրինակ՝ `state[2]`՝ ի՞նչ էր սա, flag-ի՞ տողը, թե՞ սյունը)։
- **Հեշտ ընդարձակում**։ Եթե հետագայում ուզում ես ավելացնել, օրինակ, քայլերի հաշվիչ, պարզապես ավելացնում ես `"moves"` բանալին։ List-ում ստիպված կլինեիր փոխել ինդեքսներ։

**Հարց** — Իսկ ինչու ոչ `class`. Կարող էինք, և ավելի «ճիշտ» կլիներ OOP իմաստով։ Բայց dict-ը Python-ում սովորականորեն բավարար է մի փոքրիկ խաղի համար, և դեռ չենք սովորել class-եր։

## 3. `create_world_board` — տախտակի կառուցումը

```python
def create_world_board(height, length, state):
    board = [[' ' for _ in range(length)] for _ in range(height)]
```

Քայլերը նույնն են, ինչ Java-ում.

1. **Դատարկ տախտակ ստեղծել** — բոլոր վանդակները `' '`։
2. **Խաղացողին `@` դնել** պատահական դիրքում։
3. **Դրոշակը `&` դնել** պատահական **դատարկ** վանդակում (loop՝ որ չհամընկնի խաղացողի հետ)։
4. **Կանչել `place_rocks`-ը**։

### Python-ին հատուկ բաներ

**List comprehension տախտակի ստեղծման համար**.

```python
board = [[' ' for _ in range(length)] for _ in range(height)]
```

Java-ում մենք երկու `for` ցիկլ էինք գրում.

```java
for (int i = 0; i < height; i++) {
    for (int j = 0; j < length; j++) {
        board[i][j] = ' ';
    }
}
```

Python-ում մեկ տողով անում ենք նույնը։ Սա idiomatic Python է, ուստի օգտագործում ենք։

**Թակարդ, որից խուսափեցինք**. չենք գրել `[[' '] * length] * height`, քանի որ դա կստեղծեր **նույն** ներքին list-ի **height** հատ հղում։ Փոխելով `board[0][0]`-ն կփոխեիր նաև `board[1][0]`-ն, `board[2][0]`-ն և այլն։ Շատ տարածված Python-ի սխալ է։

**`_` փոփոխական**. Երբ ցիկլում չենք օգտագործում ինդեքսը, գրում ենք `_`։ Սա համայնքային համաձայնեցում է. «այս արժեքն ինձ պետք չէ»։

**`random.randint(0, height - 1)`**. Ուշադրություն. `random.randint`-ը **ներառում է** երկու ծայրերն էլ։ Տարբերվում է Java-ի `random.nextInt(height)`-ից, որը [0, height) է (բացառելով height-ը)։ Ուստի գրում ենք `randint(0, height - 1)`, ոչ թե `randint(0, height)`։

## 4. `place_rocks` — նույն «փորձիր ու հետ կանչիր» ալգորիթմը

Տրամաբանությունը Java-ի տարբերակի հետ նույնական է, ուստի այստեղ չենք կրկնում ամբողջ բացատրությունը (տես `Labyrinth-doc.md` բաժին 4)։ Ալգորիթմի համառոտ ձև.

```
placed = 0
while placed < 10:
    ընտրիր պատահական դատարկ (r, c)
    դիր '#'
    եթե ճանապարհ դեռ կա @ -> &:
        placed += 1
    հակառակ դեպքում:
        հետ վերցրու, դարձրու ' '
```

Python-ին հատուկ նրբերանգ չկա այստեղ։ Կոդը գրեթե տող առ տող համապատասխանում է Java-ի տարբերակին։

## 5. `path_exists` — DFS-ը Python-ով

Ալգորիթմը նույնն է՝ stack-ով DFS, `visited` 2D list, 4 ուղղություն։

### Ինչն է ավելի կարճ Python-ում

**Stack-ը պարզ `list` է**.

```python
stack = [(start_row, start_col)]
...
r, c = stack.pop()
...
stack.append((nr, nc))
```

Java-ում մենք երկու զուգահեռ `int[]` ունեինք (`stackR`, `stackC`) և ձեռքով կառավարում էինք `top` ինդեքսը։ Python-ում `list`-ը արդեն ունի `append` և `pop`, որոնք O(1)-ով ավելացնում/հանում են վերջից։ Stack-ը հենց list-ն է։

**Tuple unpacking**.

```python
r, c = stack.pop()
```

Java-ում մենք առանձին-առանձին վերցնում էինք. `int r = stackR[top]; int c = stackC[top];`։ Python-ում մեկ տողով։

**Հարևանների մատրիցան որպես tuple-ների tuple**.

```python
for dr, dc in ((-1, 0), (1, 0), (0, -1), (0, 1)):
```

Կարդացվում է ուղղակիորեն. «չորս ուղղություն, յուրաքանչյուրը (շարքի շեղում, սյան շեղում)»։

### Ինչու չենք օգտագործել ռեկուրսիա

Նույն պատճառով, ինչ Java-ում. ձեռքով stack-ով ալգորիթմն ավելի թափանցիկ է սովորողի համար։ Կա նաև Python-ի լրացուցիչ պատճառ. **Python-ը խստագույնս սահմանափակում է ռեկուրսիայի խորությունը** (լռելյայն 1000)։ 10×20 տախտակի համար բավարար է, բայց դարձնում է ռեկուրսիվ լուծումը մի փոքր փխրուն։ Stack-ով տարբերակը չի կախված այդ սահմանից։

### Ինչու չենք օգտագործել BFS (`collections.deque`)

Մեզ կարճ ճանապարհ պետք չէ՝ միայն գոյությունն ենք ստուգում։ Ուստի DFS-ը (stack) ու BFS-ը (queue) կաշխատեն հավասարապես։ DFS-ն ընտրեցինք Java-ի տարբերակի հետ համահունչության համար։

## 6. `move_up/down/left/right` + `try_move`

Չորս ուղղության ֆունկցիաները միայն կանչում են `try_move`-ը նոր կոորդինատներով։ Տրամաբանությունը նույնն է.

1. Արդյոք դուրս է տախտակից → մնում ենք, տպում ենք հաղորդագրություն։
2. Արդյոք `#` ժայռ է → մնում ենք, «A rock blocks your way»։
3. Արդյոք `&` դրոշակ է → տեղափոխվում ենք, `state["won"] = True`։
4. Հակառակ դեպքում → սովորական տեղափոխություն։

### Python-ի մանր նրբերանգ

`state["won"] = True` — Java-ում օգտագործում էինք `state[4] = 1`, որովհետև `int[]`-ի մեջ boolean գրել չէինք կարող։ Python-ում `dict`-ի արժեքները տիպ չունեն սահմանված, ուստի ուղղակի `True`/`False` ենք պահում։ Շատ ավելի կարդալի.

```python
while not state["won"]:  # Python
while (state[4] == 0):   # Java
```

## 7. `print_world` — տպելու մասը

```python
def print_world(board):
    length = len(board[0])
    print('+' + '-' * length + '+')
    for row in board:
        print('|' + ''.join(row) + '|')
    print('+' + '-' * length + '+')
```

Python-ում տողերի կրկնապատկումը ուղղակիորեն աշխատում է. `'-' * length` ստեղծում է `length` երկարությամբ տող (օրինակ `'-' * 5 == '-----'`)։ Java-ում ստիպված էինք for-ով տպել։

`''.join(row)` — `row` list-ի բոլոր նիշերը միացնում է մեկ տողի՝ առանց բաժանարարի։ Java-ի `StringBuilder`-ի կամ for ցիկլով ձեռքով կպցնելու համարժեքն է, բայց մեկ ֆունկցիայի կանչով։

## 8. `input_direction` — մուտքի վավերականացում

```python
def input_direction():
    while True:
        line = input("Direction (w/a/s/d): ").strip()
        if line == "":
            print("Empty input is not allowed.")
            continue
        ch = line[0].lower()
        if ch in ('w', 'a', 's', 'd'):
            return ch
        print("Use only w, a, s or d.")
```

Java-ի `Scanner`-ի համարժեքը Python-ում `input()` ֆունկցիան է։ `input()`-ն արդեն հանում է վերջի `\n`-ը, ուստի մեզ ընդամենը պետք է `strip()` անել՝ առաջի ու վերջի բացատներից պաշտպանվելու համար։

`line[0].lower()` — Java-ի `Character.toLowerCase(line.charAt(0))`-ի համարժեքն է, բայց շատ ավելի կարճ։ Տողերի վրա մեթոդները Python-ում ուղղակի աշխատում են, և `[0]` ինդեքսացիան վերցնում է առաջին նիշը։

`ch in ('w', 'a', 's', 'd')` — Python-յան փոքր գեղեցկություն. tuple-ում անդամակցության ստուգում։ Java-ի 4 `||` պայմանների տեղ ունենք մեկ արտահայտություն.

```python
if ch in ('w', 'a', 's', 'd'):     # Python
if (ch == 'w' || ch == 'a' || ...)  // Java
```

## 9. `labyrinth` — հոսքի ուղեղը

```python
def labyrinth():
    state = {
        "player_row": 0,
        "player_col": 0,
        "flag_row": 0,
        "flag_col": 0,
        "won": False,
    }

    print_user_manual()
    board = create_world_board(10, 20, state)
    print_world(board)

    while not state["won"]:
        direction = input_direction()
        if   direction == 'w': move_up(board, state)
        elif direction == 's': move_down(board, state)
        elif direction == 'a': move_left(board, state)
        elif direction == 'd': move_right(board, state)

        print_world(board)

        if state["won"]:
            print("You reached the flag. You win!")
```

Կառուցվածքը նույնն է, ինչ Java-ում։ Տարբերությունները միայն լեզվական.

- `while not state["won"]` փոխարեն `while (state[4] == 0)`։
- `if/elif/elif` փոխարեն `if/else if/else if` (Python-ում `else if`-ը կարճացված է որպես `elif`)։

## 10. `if __name__ == "__main__":` — ինչ է սա

```python
if __name__ == "__main__":
    labyrinth()
```

Java-ում `main` մեթոդը մուտքի կետն է. JVM-ը կանչում է հենց իրեն։ Python-ում ֆայլը պարզապես **կատարվում է վերևից ներքև**։ Եթե ուղղակի գրեիր `labyrinth()` ֆայլի վերջում, այն կաշխատեր ամեն անգամ, երբ ֆայլն import արվեր այլ ֆայլի մեջ։

`__name__`-ը հատուկ փոփոխական է. երբ ֆայլը գործարկվում է **ուղիղ** (`python3 labyrinth.py`), `__name__`-ի արժեքն է `"__main__"`։ Երբ import է արվում որպես մոդուլ, `__name__`-ն ստանում է մոդուլի անունը։

Ուստի այս ստուգումը նշանակում է. «եթե ֆայլը գործարկվել է ուղիղ, սկսիր խաղը; եթե import է եղել՝ ֆունկցիաները հասանելի դարձրու, բայց ոչինչ չմեկնարկիր»։ Սա Python-ի շատ տարածված համաձայնեցում է։

## 11. Ինչու հենց այս կառուցվածքը

Նույն սկզբունքները, ինչ Java-ի փաստաթղթում.

- **Մեկ ֆունկցիա՝ մեկ պատասխանատվություն**։ Փոքրիկ, թափանցիկ ֆունկցիաներ։
- **Ճանապարհի երաշխիք**։ Ինվարիանտը (path_exists) պահպանվում է ամեն ժայռի դնելու քայլում։
- **State-ը մեկ տեղում**։ `dict`-ով, որ բոլոր ֆունկցիաները կարողանան փոխել։
- **Մուտքը պաշտպանված է**։ `input_direction`-ից դուրս գալիս միշտ ստանում ենք վավեր նիշ։

Տարբերությունն այն է, որ Python-ի շնորհիվ **կոդը զգալիորեն կարճ է** ու հաճախ ավելի կարդալի։ List comprehension-ներ, tuple unpacking, `in` օպերատոր, `dict` որպես struct — այս ամենը խնայում են տողեր ու սխալներ։

## 12. Հնարավոր հաջորդ քայլեր

- Փոխարինել `dict`-ը `dataclass`-ով (`from dataclasses import dataclass`)։ Կդառնա ավելի «type-safe», IDE-ն կօգնի։
- Քայլերի հաշվիչ (նոր բանալի dict-ում)։
- Տախտակի չափը խնդրել օգտատիրոջից։
- Բազմաթիվ դրոշակներ կամ մի քանի մակարդակ։
- Փոխարինել DFS-ը BFS-ով օգտագործելով `collections.deque` — հնարավորություն կտա ցույց տալ նաև խաղացողին **ամենակարճ** ճանապարհը։
