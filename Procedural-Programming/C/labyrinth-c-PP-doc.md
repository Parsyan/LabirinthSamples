# labyrinth.c — բացատրություն

Այս փաստաթուղթը նկարագրում է, թե ինչպես է կառուցված `labyrinth.c`-ն, ինչու հենց այդպես, և ինչպես են աշխատում ալգորիթմները։ Java-ի տարբերակը կա, ուստի այստեղ կենտրոնանում ենք **C-ին հատուկ բաների** վրա՝ pointer-ներ, struct-ներ, ձեռքով memory layout, fixed-size array-ներ, `fgets` ընդդեմ Java-ի `Scanner`-ի։

C-ն ավելի «մերկ» լեզու է, քան Java-ն. շատ բաներ, որ Java-ն թաքցնում էր (օրինակ՝ array-ի չափը, string-ի վերջը, EOF-ի դեպքերը), այստեղ պետք է ձեռքով ենք կարգավորում։ Բայց դա հենց C-ի սովորելու արժեքն է. տեսնում ես, թե իրականում ինչ է կատարվում։

---

## 1. Ընդհանուր գաղափարը

Խաղի կանոնները նույնն են, ինչ Java-ի տարբերակում. 10×20 տախտակ, `@/&/#`, `w/a/s/d`, ճանապարհի երաշխիք։

Հոսքը.

```
ձեռնարկ ցույց տալ  →  տախտակ գեներացնել  →  տպել  →
   ↓
   while չհաղթել:
       ուղղություն խնդրել  →  շարժվել  →  տպել  →  ստուգել՝ հաղթե՞լ ենք
```

Տարբերությունները լեզվից են. C-ում չկան dict-եր, ոչ class-ներ, ոչ string-ի «native» տիպ, ոչ վերին մակարդակի ֆունկցիաներ։ Ստիպված ենք օգտագործել struct-ներ և pointer-ներ։

## 2. Ինչու fixed size՝ `#define HEIGHT 10`

```c
#define HEIGHT 10
#define LENGTH 20
#define ROCK_COUNT 10
```

C-ում 2D array-ի **իրական** dynamic չափն ստանալը ավելի բարդ է, քան Java-ում։ Կան երկու հիմնական մոտեցում.

1. **Fixed size** — array-ի չափը հայտնի է կազմման ժամանակ (compile time): `char board[10][20]`։ Հարմար ու պարզ, բայց չես կարող ընթացքում փոխել։
2. **Dynamic allocation** — `malloc`-ով հատկացրած հիշողություն, `char **board` pointer-ով։ Ճկուն, բայց պետք է հիշել `free` անել, և կոդի կարդալիությունը տուժում է։

Սովորողի համար ընտրեցինք **fixed size**-ը։ Պատճառները.

- Կոդը ավելի կարդալի է. `board[i][j]` պարզ ինդեքսացիա է։
- Հիշողության կառավարման սխալներ չկան (memory leaks, double free, dangling pointers)։
- Stack-ի վրա է, ոչ heap-ի. ավելի արագ ու հեշտ դիբագելու։

Եթե հետագայում ուզում ենք փոխել տախտակի չափը, փոխում ենք `#define`-ները ու վերակազմում։ Իրականությունն այնպիսին է, որ խաղի համար 10×20-ը մշտապես բավարար է։

`#define` ընդդեմ `const int`-ի. C-ում `const int` որոշ դեպքերում չի համարվում «compile-time constant expression», ուստի array-ի չափ որպես չի աշխատի (հատկապես հին ստանդարտներում)։ `#define`-ը պարզ preprocessor-ի փոխարինում է, որը միշտ աշխատում է։ Սա C-ին հատուկ դետալ է, որից Java-ում ազատ էինք։

## 3. State-ը որպես struct, փոխանցված որպես pointer

Java-ում state-ը `int[]` էր (հղումով գնում էր)։ Python-ում՝ dict։ JavaScript-ում՝ object։ C-ում մեր ընտրությունը struct է.

```c
typedef struct {
    int player_row;
    int player_col;
    int flag_row;
    int flag_col;
    int won;   /* 0 = no, 1 = yes */
} State;
```

`typedef`-ով անունը կարճացնում ենք. գրում ենք `State`, ոչ թե `struct State`։

### Ինչու pointer-ով ենք փոխանցում

C-ում **բոլոր ֆունկցիաների փաստարկները փոխանցվում են պատճենով**։ Եթե գրեիր.

```c
void try_move(State state) {  // SUBTLE BUG: state-ը կրկնվում է
    state.player_row = 5;     // արտաքին state-ը չի փոխվում
}
```

Ֆունկցիան կստանար state-ի **պատճենը**, և բոլոր փոփոխությունները կկորչեն ֆունկցիայից վերադառնալիս։

Լուծումը. ֆունկցիային ուղարկում ենք struct-ի **հասցեն**.

```c
void try_move(State *state) {
    state->player_row = 5;   // արտաքին state-ը փոխվում է
}
```

- `State *state` — «pointer to State», այսինքն փոփոխական, որը պարունակում է State-ի հասցեն հիշողության մեջ։
- `state->player_row` — pointer-ի միջոցով հասանելի դարձնել struct-ի դաշտը։ Համարժեք է `(*state).player_row`-ին, պարզապես ավելի կարդալի։

**Հղում Java-ի հետ**. Java-ում երբ object-ի reference ենք փոխանցում, փաստորեն ճիշտ նույն բանն է կատարվում. ֆունկցիան ստանում է reference-ի պատճենը, բայց երկուսն էլ ցույց են տալիս միևնույն օբյեկտին։ Java-ի syntax-ը ուղղակի թաքցնում է pointer-ները. C-ում ստիպված ենք դրանք ուղիղ տեսնել։

## 4. Board-ը՝ 2D array որպես ֆունկցիայի փաստարկ

```c
void create_world_board(char board[HEIGHT][LENGTH], State *state)
```

Ինչու հենց այսպես.

C-ում երբ ֆունկցիային ուղարկում ես array, փաստարկի «առաջին չափը» կարող է բաց թողնվել (`board[][LENGTH]`-ն ու `board[HEIGHT][LENGTH]`-ը նույն բանն են ֆունկցիայի համար)։ Բայց **երկրորդ ու հաջորդ չափերը պետք է իմանա կոմպիլյատորը**, որպեսզի կարողանա հաշվել ինդեքսացիան՝ `board[i][j]` → `i * LENGTH + j` հիշողության մեջ։

Ուստի մենք լուռ ապավինում ենք, որ `HEIGHT` ու `LENGTH` macros-ները կազմման ժամանակ հասանելի են բոլոր ֆայլերում։ Քանի որ ամեն ինչը մեկ `.c` ֆայլում է, դա պրոբլեմ չէ։

Array-ները C-ում **չեն կրում իրենց չափը**։ Java-ում կարող էինք գրել `board.length` ու ստանալ չափը, C-ում չենք կարող։ Ուստի `HEIGHT`, `LENGTH` macros-ները փաստացի մեր «չափի» աղբյուրն են ողջ ծրագրի համար։

## 5. `create_world_board` — տախտակի կառուցումը

Քայլերն ու տրամաբանությունը նույնն են, ինչ Java-ում.

1. Բոլոր վանդակները `' '`-ի լցնել։
2. Խաղացողին `@` պատահական դիրքում։
3. Դրոշակը `&` պատահական դատարկ վանդակում։
4. Կանչել `place_rocks`-ը։

### `rand() % HEIGHT` — ինչու սա

```c
int player_row = rand() % HEIGHT;
```

C-ի `rand()`-ը վերադարձնում է `int` 0-ից մինչև `RAND_MAX` (սովորաբար շատ մեծ թիվ)։ Այն կտրելու համար `HEIGHT` շրջանակին, օգտագործում ենք modulo-ն (`%`)։

**Մանր թերություն**. այս մոտեցումը մի փոքր «շեղված» (biased) է, եթե `RAND_MAX`-ը հավասարաչափ չի բաժանվում `HEIGHT`-ի վրա։ Բայց փոքր թվերի համար (10, 20) շեղումն աննշան է, ուստի սովորողի համար `rand() % HEIGHT`-ը լրիվ բավարար է։

### `srand((unsigned int) time(NULL))` — սերմ

`main`-ում.

```c
srand((unsigned int) time(NULL));
```

`rand()`-ը deterministic է. տալիս է նույն հաջորդականությունը ամեն գործարկման ժամանակ, քանի դեռ սերմը (seed) նույնն է։ Default-ով սերմը 1 է։ Ուրեմն առանց `srand`-ի, ամեն խաղ կլիներ նույն տախտակը։

`time(NULL)`-ը վերադարձնում է ընթացիկ ժամանակը վայրկյաններով (Unix timestamp)։ Որպես սերմ օգտագործելով, ամեն գործարկում տարբեր սերմ է, ուստի՝ տարբեր տախտակ։

`(unsigned int)`-ի cast-ը. `time(NULL)`-ը վերադարձնում է `time_t`, որը կարող է լինել ավելի մեծ, քան `unsigned int`-ը։ Cast-ով հստակորեն ասում ենք. «վերցրու ստորին bits-երը»։

C-ին հատուկ նրբերանգ չկա, ալգորիթմը գործում է ուղիղ։

## 7. `path_exists` — DFS-ը C-ով

Նույն ալգորիթմը. stack, visited, 4 ուղղություն։ Բայց C-ում **stack-ը պատրաստի չկա**։ Ուստի ձեռքով ենք պատրաստում.

```c
int stack_r[HEIGHT * LENGTH];
int stack_c[HEIGHT * LENGTH];
int top = 0;
stack_r[top] = start_row;
stack_c[top] = start_col;
top++;
```

Երկու զուգահեռ array (մեկը՝ տողերի, մյուսը՝ սյուների համար) + `top` ինդեքս, որը ցույց է տալիս «հաջորդ ազատ տեղը»։

**Ինչու երկու array, ոչ թե array of pairs**. C-ում `struct {int r; int c;}` ստեղծել ու array-ի մեջ դնելը հնարավոր է, բայց ավելացնում է կոդի բարդություն։ Երկու զուգահեռ array-ները շատ տարածված ձև են և ուղղակի կարդալի։

**Visited-ի մասին**.

```c
int visited[HEIGHT][LENGTH];
for (int i = 0; i < HEIGHT; i++)
    for (int j = 0; j < LENGTH; j++)
        visited[i][j] = 0;
```

C-ում local array-ները **սկզբնական արժեք չունեն**. դրանք պարունակում են «աղբ» (garbage)՝ ինչ էլ որ եղել է հիշողության այդ հատվածում։ Ուստի պարտադիր է ձեռքով 0-ով լցնել։

Կարող էինք գրել `int visited[HEIGHT][LENGTH] = {0};` — սա հատուկ initializer է, որ լցնում է բոլոր էլեմենտները 0-ով։ Բայց ձեռքով `for`-ով ավելի թափանցիկ է սովորողի համար։

### Ինչու stack-ի չափը հենց `HEIGHT * LENGTH`

Որովհետև ամենավատ դեպքում մենք կարող ենք visit անել ամեն վանդակ ճիշտ մեկ անգամ (visited-ի շնորհիվ), ուստի stack-ում առավելագույնը կլինի `HEIGHT * LENGTH` էլեմենտ։ Մեծացնելու հարկ չկա, փոքրացնելու հնարավորություն չկա առանց ալգորիթմը խախտելու։

### Static-ի օգտագործումը խուսափել

Stack-ն ու visited-ը local array-ներ են, ուստի ֆունկցիայից վերադառնալիս ինքնաբերաբար անհետանում են։ Չենք օգտագործել `static`, քանի որ չենք ուզում նրանք մնան երկար ընթացքում՝ տեղ զբաղեցնելով։

## 8. `try_move` — ինչու pointer-ով

```c
void try_move(char board[HEIGHT][LENGTH], int new_row, int new_col, State *state)
```

- `board` — 2D array, փոխանցվում է որպես հղում (C-ում array-ները default-ով այդպես են փոխանցվում)։
- `new_row`, `new_col` — պարզ int-եր, պատճենով։ Մեզ պետք չէ ֆունկցիայի ներսից փոխել դրանք։
- `state` — pointer to State, որպեսզի կարողանանք փոխել `state->won`-ը, `state->player_row`-ը։

Ստուգումները նույնն են. սահման → ժայռ → դրոշակ → սովորական։

## 9. `print_world` — `printf`-ով տպելը

```c
printf("+");
for (int j = 0; j < LENGTH; j++) printf("-");
printf("+\n");
```

C-ում string-ի կրկնապատկման օպերատոր չկա (`'-' * 20` չի աշխատի)։ Ձեռքով `for` ենք գրում։

Կարող էինք օգտագործել format specifier. `printf("%.*s", LENGTH, "--------------------")` — սա կաշխատի, բայց ավելի շփոթիչ է սովորողի համար։

`%c` format specifier-ով ենք տպում նիշերը.

```c
printf("%c", board[i][j]);
```

`putchar(board[i][j])`-ն էլ նույնն է անում, մի փոքր ավելի արագ։ Բայց `printf` օգտագործում ենք համապատասխանության համար (նույն ֆունկցիան ողջ ֆայլում)։

## 10. `input_direction` — `fgets` ընդդեմ `scanf`-ի

C-ում մուտքի կարդալու մի քանի ձև կա.

- `scanf("%c", &ch)` — մեկ նիշ ստանում է, բայց buffer-ում թողնում է `\n`-ը, որ հաջորդ կարդալիս խնդիր է առաջացնում։
- `scanf("%s", buffer)` — բառ կարդում է մինչև բացատը, բայց բերում է buffer overflow-ի վտանգ եթե զգույշ չես։
- `fgets(buffer, sizeof(buffer), stdin)` — մեկ ամբողջ տող կարդում է, ներառյալ `\n`-ը, ապահով չափի սահմանով։ **Ընտրեցինք սա**։

```c
char buffer[128];
while (1) {
    printf("Direction (w/a/s/d): ");
    fflush(stdout);
    if (fgets(buffer, sizeof(buffer), stdin) == NULL) {
        printf("\nGoodbye.\n");
        exit(0);
    }
    ...
}
```

### `fflush(stdout)` — ինչու

C-ի `stdout`-ը buffered է. տպած տեքստը կարող է փաստացի էկրանին չհայտնվել մինչև `\n` նիշը կամ buffer-ը լցվելը։ Մեր prompt-ը (`Direction (w/a/s/d): `) վերջում `\n` չունի, ուստի առանց `fflush`-ի օգտատերը կտեսնի դատարկ էկրան և չի հասկանա, որ պետք է ինչ-որ բան մուտքագրի։

`fflush(stdout)` ստիպում է buffer-ը անմիջապես «լցել» էկրանին։

### EOF-ի դեպքը

`fgets`-ը վերադարձնում է `NULL` երբ ֆայլի վերջն է հասել (օրինակ՝ Ctrl+D Linux-ում, Ctrl+Z Windows-ում)։ Մենք օգտվում ենք այս սիգնալից և ծրագիրը մաքուր փակում ենք `exit(0)`-ով։

**Կարևոր սխալ, որից խուսափեցինք**. եթե `continue` անեինք NULL-ի դեպքում, ծրագիրը կլցվեր անվերջ loop-ի մեջ (քանի որ `fgets`-ը միշտ NULL կվերադարձներ, ստուգում էինք էմպիրիկ կերպով). հենց դա էր նախնական bug-ը, որը հայտնաբերեցինք testing-ի ընթացքում։

### Տողի մաքրումը

```c
int i = 0;
while (buffer[i] == ' ' || buffer[i] == '\t') i++;

if (buffer[i] == '\n' || buffer[i] == '\0') {
    printf("Empty input is not allowed.\n");
    continue;
}

char ch = (char) tolower((unsigned char) buffer[i]);
```

- Բացատների բաց թողնում առաջից (`while` loop)։
- Եթե հասանք `\n`-ի կամ `\0`-ի, տողը դատարկ էր։
- `tolower`-ը մեծատառը դարձնում է փոքրատառ։ `(unsigned char)`-ի cast-ը C-ի specific պահանջ է. `tolower`-ը սահմանված է միայն `unsigned char` և EOF արժեքների համար, իսկ `char`-ը կարող է լինել signed (բացասական արժեքով՝ եթե բարձր bit-ը 1 է)։ Cast-ով խուսափում ենք undefined behavior-ից։

## 11. `labyrinth` — հոսքի ուղեղը

```c
void labyrinth(void) {
    State state = {0, 0, 0, 0, 0};
    char board[HEIGHT][LENGTH];

    print_user_manual();
    create_world_board(board, &state);
    print_world(board);

    while (state.won == 0) {
        char dir = input_direction();
        ...
        print_world(board);
        if (state.won == 1) {
            printf("You reached the flag. You win!\n");
        }
    }
}
```

Մի քանի կարևոր C-ին հատուկ բան.

- **`State state = {0, 0, 0, 0, 0}`** — initialize ենք ամեն ինչը 0-ով։ Առանց դրա պարունակությունը կլիներ աղբ։ Կարող էինք գրել նաև `State state = {0};` (հատուկ initializer, որ լցնում է մնացածը 0-ով), բայց բացահայտ թվարկումն ավելի թափանցիկ է։
- **`&state`** — ուղարկում ենք state-ի **հասցեն**, քանի որ ֆունկցիան սպասում է `State *`։ `&` օպերատորը վերցնում է փոփոխականի հասցեն։
- **`board`-ը պարզապես անվանումով ենք ուղարկում**, ոչ `&board`։ Որովհետև C-ում array-ի անունն ինքնին «քայքայվում է» (decay) իր առաջին էլեմենտի pointer-ի։ Սա C-ի յուրահատկություն է. array-ները ֆունկցիաների միջոցով միշտ ուղարկվում են որպես pointer, անկախ նրանից թե ինչ syntax օգտագործում ես։

## 12. Forward declarations — ինչու դրանք պետք են

Ֆայլի վերևում ունենք.

```c
void create_world_board(char board[HEIGHT][LENGTH], State *state);
void place_rocks(char board[HEIGHT][LENGTH], State *state);
int  path_exists(char board[HEIGHT][LENGTH], State *state);
/* ... */
```

C-ի կոմպիլյատորը ֆայլը կարդում է **վերևից ներքև**։ Եթե `create_world_board`-ը կանչում է `place_rocks`-ը, բայց `place_rocks`-ը դեռ չի սահմանվել ֆայլում, կոմպիլյատորը չի կարող ստուգել, թե արդյոք կանչը ճիշտ է։

**Forward declaration**-ները լուծում են այս խնդիրը. դրանք ասում են «այս ֆունկցիան գոյություն ունի, ահա իր signature-ը, մարմինն ավելի ուշ կգա»։ Java-ում սա պետք չէր, որովհետև Java-ի կոմպիլյատորն ամբողջ class-ը նախ-պես ստուգում է։

Կարող էինք պարզապես **դասավորել ֆունկցիաները հակառակ կարգով** (ամենացածր մակարդակը՝ վերևում, ամենաբարձրը՝ ներքևում)։ Բայց forward declarations-ով կարող ենք ֆունկցիաները դասավորել ինչպես **տրամաբանական է կարդալու համար**, ոչ թե ինչպես կպահանջեր ստուգման կարգը։

## 13. Ինչու հենց այս կառուցվածքը

Նույն սկզբունքները.

- **Մեկ ֆունկցիա՝ մեկ պատասխանատվություն**։
- **Ճանապարհի երաշխիք**։
- **State-ը մեկ struct-ում**, ոչ թե բազմաթիվ global փոփոխականներով։
- **Մուտքը պաշտպանված է**, ներառյալ EOF-ի դեպքը։

C-ին հատուկ ընտրությունները.

- **Fixed-size array-ներ** dynamic allocation-ի փոխարեն — պարզ ու ապահով։
- **Pointer-ով state** struct-ի փոփոխման համար (C-ի pass-by-value-ի շուրջ)։
- **`fgets`** մուտքի կարդալու համար — ապահով չափով, ներառյալ EOF-ի մշակումը։
- **Forward declarations** — որպեսզի ֆունկցիաների կարգը կարդալիության հիմքով լինի։

## 14. Հնարավոր հաջորդ քայլեր

- **Dynamic size**. Փոխարինել `#define HEIGHT 10`-ը պարամետրով, որ օգտատերը խնդրի սկզբից։ Կպահանջի `malloc` և `free`, և bound-ները փոխանցել ամեն ֆունկցիային։
- **Աղյուսակը գույներով**. ANSI escape codes-ով (`\033[31m` և այլն) կարող ենք ժայռերը դարձնել կարմիր, դրոշակը՝ կանաչ։
- **Քայլերի հաշվիչ** State-ում։
- **Multiple levels** — `Level *` array-ով։
- **Tests** — ալգորիթմները (`path_exists`, `place_rocks`) թեստավորելու համար։
